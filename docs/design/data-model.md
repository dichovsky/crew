# crew Data Model

This document defines the binding v1 contract for how crew stores data, plus conventions
added after v1 on top of the unchanged schema. The SQL names below are for implementers; the
surrounding prose uses the domain vocabulary from [CONTEXT.md](../../CONTEXT.md).

## Store location and opening

The State Store — the SQLite database every Agent in a Workspace shares — is
`<workspace>/.crew/state/crew.db`. Each command opens it once, with:

- Node.js `DatabaseSync` timeout `5000` ms (mirrored by `PRAGMA busy_timeout = 5000`);
- extension loading denied at open (`allowExtension: false`);
- `PRAGMA foreign_keys = ON`;
- defensive mode on: `PRAGMA trusted_schema = OFF` and `PRAGMA cell_size_check = ON`;
- `PRAGMA journal_mode = WAL` (write-ahead logging, an SQLite mode that lets readers keep
  working while one writer writes) and `PRAGMA synchronous = NORMAL`;
- timestamps as integer epoch seconds (seconds since 1970), generated by the calling code
  from a single clock reading per operation.

The database path must be on a local filesystem; network shares are unsupported. Every SQL
statement passes user input as bound parameters, never by pasting it into the SQL text.

`<workspace>` is found by walking up from the working directory to the nearest folder that
contains `.crew/` — every command and every launched pane discovers it the same way. A Crew
launched inside a git worktree (a separate working copy of the repository sharing the same
history) therefore resolves the *worktree's* own `.crew/` and opens a fresh Store that is
**local to that worktree and lives only as long as it does**; this is deliberate — it does not
redirect to a store shared with the main repository (ADR-0011).

## Schema version 7 (current)

The SQL below ("DDL" — the statements that define the database's shape) is binding in shape;
migration code may split it into separate statements.
Version 2 adds the nullable `launch_token` column and its partial index; every
other object is unchanged from version 1. Version 3 adds the `abandoned_at` column to
`tasks`, extends the `status` CHECK and the per-status rules to cover the new
`'abandoned'` final state, and extends the `task_events` CHECKs to allow the
`'abandoned'` event type and transition. Version 4 adds the nullable
`worktree_path`/`worktree_branch`/`worktree_base_ref` trio to `tasks` — the on-disk
worktree the Task's assignee works in, tracked independently of `status` (created at
start, still present through `submitted` and `completed`, cleared only by a later land
or abandon action), which is why the trio carries its own standalone `CHECK` instead of
joining the per-status rule. `worktree_base_ref` stores the ref the Task's branch was
actually created from, resolved to a concrete branch name (never the literal `"HEAD"`):
a later "has this genuinely landed" check must compare the branch against a fixed
starting point, not against whatever `HEAD` happens to point to when the check runs from
some other working directory. Version 4 also adds the new `review_worktrees` table — one
row per reviewing Agent, describing that Agent's own reusable worktree, used to actually
check out a Task's branch for review. Version 5 adds the single-row `agent_mutations`
table — a counter that only ever increases — and three triggers on `agents` that bump it
on every INSERT, UPDATE, and DELETE. The reason: some agent-row changes move
none of the `MAX`/`COUNT` values the Console's change signature is built from — a
launch-teardown cleanup deleting a row that was not the newest, or a `last_seen` update
landing in the same clock second — yet the Console change signature must still move for
them. Version 5 also drops `idx_task_events_task`: it repeated, column for
column, the index SQLite already builds internally for the `UNIQUE (task_id, revision)`
constraint, so every Task transition was maintaining two identical index structures
that offered no extra way to read the data. Version 6 extends the `messages.kind` CHECK
with the structured `clear_safe` Sign-off value (ADR-0016) — the Message kind
that tells a Worker its Task has fully landed — created only by the land and abandon
transitions. Version 7 adds the `observable_mutations` counter and triggers covering
every Message, Task, and Task Event write, so the Console's live update stream (SSE —
server-sent events) also notices when prune deletes rows below a maximum that remains.
See the migration note below.

```sql
CREATE TABLE agents (
  id            TEXT PRIMARY KEY CHECK (length(id) BETWEEN 1 AND 64),
  role          TEXT NOT NULL CHECK (length(role) BETWEEN 1 AND 64),
  platform_id   TEXT CHECK (platform_id IS NULL OR length(platform_id) BETWEEN 1 AND 64),
  joined_at     INTEGER NOT NULL,
  last_seen     INTEGER NOT NULL,
  status        TEXT NOT NULL CHECK (status IN ('active', 'archived')),
  archived_at   INTEGER,
  launch_token  TEXT CHECK (launch_token IS NULL OR length(launch_token) BETWEEN 32 AND 128),
  CHECK (last_seen >= joined_at),
  CHECK (
    (status = 'active' AND archived_at IS NULL) OR
    (status = 'archived' AND archived_at IS NOT NULL AND archived_at >= joined_at)
  )
) STRICT;

CREATE TABLE tasks (
  id                  TEXT PRIMARY KEY,
  title               TEXT NOT NULL CHECK (length(title) BETWEEN 1 AND 500),
  body                TEXT NOT NULL DEFAULT '' CHECK (length(body) <= 100000),
  creator_id          TEXT NOT NULL REFERENCES agents(id),
  assignee_id         TEXT NOT NULL REFERENCES agents(id),
  reviewer_id         TEXT NOT NULL REFERENCES agents(id),
  status              TEXT NOT NULL CHECK (
    status IN ('queued', 'in_progress', 'submitted', 'completed', 'abandoned')
  ),
  revision            INTEGER NOT NULL DEFAULT 0 CHECK (revision >= 0),
  lease_owner_id      TEXT REFERENCES agents(id),
  lease_expires_at    INTEGER,
  submission_summary  TEXT CHECK (
    submission_summary IS NULL OR length(submission_summary) BETWEEN 1 AND 100000
  ),
  submitted_at        INTEGER,
  review_summary      TEXT CHECK (review_summary IS NULL OR length(review_summary) <= 100000),
  completed_at        INTEGER,
  abandoned_at        INTEGER,
  worktree_path       TEXT CHECK (worktree_path IS NULL OR length(worktree_path) BETWEEN 1 AND 4096),
  worktree_branch     TEXT CHECK (worktree_branch IS NULL OR length(worktree_branch) BETWEEN 1 AND 255),
  worktree_base_ref   TEXT CHECK (worktree_base_ref IS NULL OR length(worktree_base_ref) BETWEEN 1 AND 255),
  created_at          INTEGER NOT NULL,
  updated_at          INTEGER NOT NULL,
  CHECK (updated_at >= created_at),
  CHECK ((worktree_path IS NULL) = (worktree_branch IS NULL)
    AND (worktree_path IS NULL) = (worktree_base_ref IS NULL)),
  CHECK (
    (status = 'queued' AND lease_owner_id IS NULL AND lease_expires_at IS NULL
      AND submission_summary IS NULL AND submitted_at IS NULL
      AND review_summary IS NULL AND completed_at IS NULL AND abandoned_at IS NULL) OR
    (status = 'in_progress' AND lease_owner_id IS NOT NULL AND lease_expires_at IS NOT NULL
      AND submission_summary IS NULL AND submitted_at IS NULL
      AND review_summary IS NULL AND completed_at IS NULL AND abandoned_at IS NULL) OR
    (status = 'submitted' AND lease_owner_id IS NULL AND lease_expires_at IS NULL
      AND submission_summary IS NOT NULL AND submitted_at IS NOT NULL
      AND review_summary IS NULL AND completed_at IS NULL AND abandoned_at IS NULL) OR
    (status = 'completed' AND lease_owner_id IS NULL AND lease_expires_at IS NULL
      AND submission_summary IS NOT NULL AND submitted_at IS NOT NULL
      AND completed_at IS NOT NULL AND abandoned_at IS NULL) OR
    (status = 'abandoned' AND lease_owner_id IS NULL AND lease_expires_at IS NULL
      AND review_summary IS NULL AND completed_at IS NULL AND abandoned_at IS NOT NULL)
  )
) STRICT;

CREATE TABLE messages (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  sender_id     TEXT NOT NULL REFERENCES agents(id),
  recipient_id  TEXT NOT NULL REFERENCES agents(id),
  content       TEXT NOT NULL CHECK (length(content) BETWEEN 1 AND 100000),
  kind          TEXT NOT NULL DEFAULT 'note' CHECK (
    kind IN ('note', 'task_assigned', 'task_submitted', 'task_approved', 'task_requeued',
             'clear_safe')
  ),
  task_id       TEXT REFERENCES tasks(id) ON DELETE CASCADE,
  reply_to      INTEGER REFERENCES messages(id) ON DELETE SET NULL,
  created_at    INTEGER NOT NULL,
  read_at       INTEGER,
  CHECK ((kind = 'note') OR task_id IS NOT NULL),
  CHECK (read_at IS NULL OR read_at >= created_at)
) STRICT;

CREATE TABLE task_events (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  task_id       TEXT NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
  revision      INTEGER NOT NULL CHECK (revision >= 0),
  event_type    TEXT NOT NULL CHECK (
    event_type IN ('created', 'started', 'submitted', 'approved', 'requeued', 'abandoned')
  ),
  actor_id      TEXT NOT NULL REFERENCES agents(id),
  from_status   TEXT,
  to_status     TEXT NOT NULL,
  detail        TEXT NOT NULL DEFAULT '' CHECK (length(detail) <= 100000),
  created_at    INTEGER NOT NULL,
  UNIQUE (task_id, revision),
  CHECK (from_status IS NULL OR from_status IN ('queued', 'in_progress', 'submitted', 'completed', 'abandoned')),
  CHECK (to_status IN ('queued', 'in_progress', 'submitted', 'completed', 'abandoned')),
  CHECK (
    (event_type = 'created' AND revision = 0 AND from_status IS NULL AND to_status = 'queued') OR
    (event_type = 'started' AND revision > 0 AND from_status = 'queued' AND to_status = 'in_progress') OR
    (event_type = 'submitted' AND revision > 0 AND from_status = 'in_progress' AND to_status = 'submitted') OR
    (event_type = 'approved' AND revision > 0 AND from_status = 'submitted' AND to_status = 'completed') OR
    (event_type = 'requeued' AND revision > 0 AND from_status IN ('in_progress', 'submitted') AND to_status = 'queued') OR
    (event_type = 'abandoned' AND revision > 0 AND from_status IN ('queued', 'in_progress', 'submitted') AND to_status = 'abandoned')
  )
) STRICT;

CREATE TABLE review_worktrees (
  agent_id      TEXT PRIMARY KEY REFERENCES agents(id),
  path          TEXT NOT NULL CHECK (length(path) BETWEEN 1 AND 4096),
  base_ref      TEXT NOT NULL CHECK (length(base_ref) BETWEEN 1 AND 255),
  current_ref   TEXT CHECK (current_ref IS NULL OR length(current_ref) BETWEEN 1 AND 255),
  created_at    INTEGER NOT NULL,
  updated_at    INTEGER NOT NULL,
  CHECK (updated_at >= created_at)
) STRICT;

CREATE TABLE agent_mutations (
  id      INTEGER PRIMARY KEY CHECK (id = 1),
  cursor  INTEGER NOT NULL CHECK (cursor >= 1)
) STRICT;

CREATE TABLE observable_mutations (
  id      INTEGER PRIMARY KEY CHECK (id = 1),
  cursor  INTEGER NOT NULL CHECK (cursor >= 1)
) STRICT;

CREATE INDEX idx_messages_unread
  ON messages(recipient_id, id) WHERE read_at IS NULL;
CREATE INDEX idx_messages_history
  ON messages(created_at, id);
CREATE INDEX idx_messages_task
  ON messages(task_id) WHERE task_id IS NOT NULL;
CREATE INDEX idx_messages_reply_to
  ON messages(reply_to) WHERE reply_to IS NOT NULL;
CREATE INDEX idx_tasks_assignee_status
  ON tasks(assignee_id, status, updated_at);
CREATE INDEX idx_tasks_reviewer_status
  ON tasks(reviewer_id, status, updated_at);
CREATE INDEX idx_tasks_expired_lease
  ON tasks(lease_expires_at) WHERE status = 'in_progress';
CREATE INDEX idx_agents_launch_token
  ON agents(launch_token) WHERE launch_token IS NOT NULL;

CREATE TRIGGER trg_agents_mutation_insert AFTER INSERT ON agents

END;

    ON CONFLICT (id) DO UPDATE SET cursor = cursor + 1;

END;

BEGIN

END;
CREATE TRIGGER trg_messages_observable_mutation_update AFTER UPDATE ON messages
BEGIN

END;
CREATE TRIGGER trg_messages_observable_mutation_delete AFTER DELETE ON messages
BEGIN

END;
CREATE TRIGGER trg_tasks_observable_mutation_insert AFTER INSERT ON tasks
BEGIN

END;
CREATE TRIGGER trg_tasks_observable_mutation_update AFTER UPDATE ON tasks
BEGIN
```

`review_worktrees` needs no extra index: the primary key already covers the only query
used (a lookup by `agent_id`). `task_events` needs no explicit index either: the index
SQLite builds automatically for the `UNIQUE (task_id, revision)` constraint serves both
the reads by `task_id` and the reads by `(task_id, revision)` (the explicit
`idx_task_events_task` that duplicated it was dropped in version 5).
`agent_mutations` holds at most one row (`id = 1`), created on demand by the first write
to `agents`; the triggers make `cursor` only ever increase within the lifetime of one
database file, and a missing row is read as cursor 0.

## Entity invariants

An invariant is a rule that must hold at all times; the rules below hold for every row of
each entity.

### Agents

- ids are 1-64 characters and match `[A-Za-z0-9][A-Za-z0-9._-]*`;
- `@all` is reserved for broadcast syntax and is never stored;
- an archived id stays reserved: a join without `--resume` never takes it over or inherits
  its work;
- `--resume` is the only way an archived Agent becomes active again;
- resume keeps the Role, keeps the platform unless a replacement is explicitly supplied,
  keeps `joined_at`, and refreshes `last_seen`;
- leave sets `archived_at` while keeping `last_seen`;
- `last_seen` records when the Agent last did something; it never proves the process is
  still running;
- `launch_token` is set only when a brand-new Agent is created from the launcher's
  `CREW_LAUNCH_TOKEN` environment variable (never on resume). It is the proof of which
  launch created the row — the launch-teardown cleanup deletes only rows carrying its own
  token — and it is never part of a rendered Agent record (human or JSON).
- The human Operator is represented by an ordinary Agent row with id `operator` and
  `platform_id IS NULL`. This is a naming convention, not a special privileged row: the
  Operator is subject to the same active/archived checks, foreign keys, Task authority, and
  Message rules as every other Agent, and every read and write on the Operator's behalf goes
  through the existing Store methods. `crew ui` changes nothing in the State Store schema
  and needs no schema-version bump or migration.

### Messages

- a direct send requires both sender and recipient to be active at the moment the write
  commits;
- a broadcast inserts one row per other active Agent, all in one transaction;
- `reply_to`, when supplied, must point at a Message the caller sent or received; this stops
  an Agent from accidentally quoting a conversation it was never part of, even though all
  Agents in a Workspace already trust each other;
- `receive` sets `read_at` once and never deletes a row;
- Task notification Messages are created only by Task transitions, never by any freeform CLI
  flag.
- the structured `clear_safe` Sign-off (ADR-0016) — the Message telling a Worker its Task has
  fully landed and its context is safe to clear — is created only by `task land` (the
  Sign-off to the assignee) and `task abandon` (the assignee's copy; the creator/reviewer
  copies stay `note`s); like every kind other than `note`, it always carries a `task_id`;
- a successful direct or broadcast send refreshes the sender's activity with the operation
  timestamp — even a broadcast that reached nobody; receive validates its active Agent and
  refreshes it the same way even when no Message was claimed;
- Notification recipients are de-duplicated by Agent id, the Agent that performed the
  transition is left out, and archived Agents are skipped; ending up with nobody to notify
  is valid. A notification's content is a short, length-bounded pointer line; the free-text
  Submission/Review/requeue reason travels in the Task Event detail instead. Exception: the
  assignee's `clear_safe` Sign-off is delivered even when the assignee IS the acting Agent
  (ADR-0016) — it is the durable unread signal the Relay's reset logic depends on, not a
  courtesy note — so only an archived assignee suppresses it.

### Tasks

- a Task `id` is a UUIDv4 string generated by crew at create time;
- creator, assignee, and reviewer must all be active when a Task is created, and the forward
  transitions (start, submit, approve) likewise require all three to be active at the moment
  of the transition;
- assignee and reviewer may be the same Agent, though the built-in Team keeps them separate;
- an Agent may hold several Leases at once; a Lease (the self-expiring claim on an
  in-progress Task) belongs to one Task — it never locks the whole Agent;
- start requires the Task to be `queued`, the caller to be its assignee, and the expected
  revision; it sets `lease_owner_id = assignee` and
  `lease_expires_at = <operation-clock> + 900` (a 15-minute Lease);
- start also fills in `worktree_path`, `worktree_branch`, and `worktree_base_ref` when the
  workspace config enables per-Worker worktrees, leaves all three `NULL` when it does not,
  and keeps an already-recorded trio if the feature is switched off before a requeued Task
  restarts;
- submit requires the Task to be `in_progress`, the caller to hold its unexpired Lease, and
  the expected revision;
- approve requires the Task to be `submitted`, the caller to be its reviewer, and the
  expected revision;
- requeue requires the actor to be the Task's creator or reviewer, plus a non-empty reason;
  the actor and any new `--to` assignee must be active, but the previous assignee is allowed
  to be gone — otherwise a Task whose owner left could never be recovered;
- `submitted` may be requeued immediately; `in_progress` may be requeued only after the
  Lease has expired;
- abandon always clears `worktree_path`, `worktree_branch`, and `worktree_base_ref` as part
  of the status change itself; whether the later removal of the worktree from disk (which
  happens outside the transaction) succeeds does not change that;
- each successful transition increments `revision` exactly once and appends the matching
  Task Event carrying the same revision;
- a completed Task can never be modified in v1;
- `land` requires `status = 'completed'` and a non-null `worktree_path`. When the worktree
  has uncommitted changes, or its branch is not yet contained in ("an ancestor of")
  `worktree_base_ref`, land refuses and changes nothing — unless `--force` is given.
  Otherwise it removes the worktree and branch, and clears `worktree_path`,
  `worktree_branch`, and `worktree_base_ref` in the same all-or-nothing step that sends the
  Sign-off (ADR-0014) to the assignee as a structured `clear_safe` Message (ADR-0016).

### Review worktrees

- one row per reviewing Agent; `agent_id` is the primary key;
- `current_ref IS NULL` means the worktree is resting on `base_ref`; a non-NULL `current_ref`
  means it is currently checked out to a Task's branch for a review that is under way;
- the row (and its worktree) is created the first time an Agent runs `task review` and is
  never deleted — unlike a Task's own worktree (cleared by `land`/`abandon`), a reviewing
  Agent's worktree is kept and reused for every Task that Agent ever reviews.

### Console session record (non-Store)

What the Console's `GET /api/sessions` read returns (SRS FR-U35) is **not** stored in the
database at all: it is computed from the pane-map files a launch wrote under
`.crew/generated/<session>/`, combined with a live check that crew owns the tmux session. For
each live crew-owned session it emits a JSON record
`{ type: 'session', schema_version: 1, session_name, pane_count, agent_count, started_at }`
(new fields may be added later), where `started_at` is the time the pane-map was written, in
epoch seconds, on the same clock the Store uses. Only sessions whose ownership token matches
the live tmux session appear; stale, foreign, or malformed maps are left out.

## Transaction catalogue

Each row below is one database transaction: either all of its effects happen, or none do. In
the table, "CAS" means compare-and-set — the update succeeds only if the row is still at the
revision the operation read — and "touch" means refreshing an Agent's `last_seen`.

| Operation | Atomic effects |
|---|---|
| join | reserve/reactivate exactly one Agent id |
| direct send | validate active participants/reply, insert one note, touch sender |
| broadcast | validate sender, snapshot active recipients, insert all notes, touch sender |
| receive | validate/touch Agent, then mark and return at most `limit` unread Messages with one claim statement |
| task create | insert Task revision 0, `created` event, assignment Message |
| task start | CAS Task to revision +1, append `started` event |
| task submit | CAS Task, append `submitted` event, notify reviewer and creator |
| task approve | CAS Task, append `approved` event, notify creator and assignee |
| task requeue | CAS Task, clear work/review fields, append `requeued` event, notify assignee/creator/reviewer |
| prune | select retention set, delete in referential order, commit; vacuum separately |
| launch-teardown reap | delete only untouched Agent rows matching this launch's token in one immediate transaction |

Task transactions use `BEGIN IMMEDIATE` (the transaction claims the right to write up front
instead of failing later). The operation clock is read once per operation, so the Task, its
Event, and its notifications all carry the same timestamp.

The Console detects change through `Store.getChangeSignature()` (FR-U22), a read-only query
that gathers a set of counters: the highest (`MAX()`) Message id, Task-Event id, Task
`updated_at`, Agent `last_seen`, and Agent `archived_at`; the count of Tasks whose Lease has
expired (`staleLeaseCount`, FR-U33) — the only counter that depends on the current time,
judged against the operation clock; the `agent_mutations.cursor` value
(`agentMutationCursor`), which schema-v5 triggers bump on every `agents`
INSERT/UPDATE/DELETE and which covers the agent-row changes the other counters cannot see (a
cleanup's DELETE of a row that was not the newest; a `last_seen` update landing in the same
clock second); and `observable_mutations.cursor` (`observableMutationCursor`), whose
schema-v7 triggers cover every Message, Task, and Task-Event write, including a retention
delete of rows below a maximum that remains. Reading the signature is itself purely a read:
it marks nothing read and stamps no Agent activity.

## Delivery semantics

`receive` delivers each Message at most once — once, or, if the process crashes at exactly
the wrong moment, not at all; never twice:

- two receives running at the same time never both return the same Message;
- a Message committed while a receive is running stays unread unless that receive's own
  claiming write selected it;
- output is sorted by `(created_at, id)` after the database returns the claimed rows
  (`RETURNING`);
- a crash after the database commit but before the output is seen means a Message is marked
  read without anyone having read it — it disappears from the Inbox;
- history still holds the row and its `read_at`, so the content can be recovered by hand.

The wording “lost-message-free” is intentionally not used. Delivering exactly once is not
possible when the hand-off spans an SQLite commit and a terminal's standard output, unless
the consumer confirms receipt — and crew has no such acknowledgement.

## Schema migration contract

1. Read `user_version` (SQLite's built-in schema version number) before any normal
   operation.
2. If the database's version is newer than this binary understands, reject it with
   `UNSUPPORTED_SCHEMA` and exit 1.
3. Treat version 0 as a brand-new database only: create the current schema only when the file
   contains no application-defined tables, indexes, views, or triggers. A version-0 database
   that is not empty is rejected with `UNSUPPORTED_SCHEMA` and left untouched.
4. For an older supported *released* version (non-zero), take an exclusive transaction for
   the migration, so nothing else can write while it runs.
5. Apply the migrations in order. Before each change, confirm the expected tables and columns
   are present by asking SQLite's own metadata (`PRAGMA table_info`, `sqlite_schema`) — never
   by matching the text of SQLite error messages.
6. Run `PRAGMA foreign_key_check`, and update `user_version` only at the very end.
7. On any failure, roll the whole migration back.

A freshly created `DatabaseSync` reports `user_version` 0; creating the current schema sets
the version straight to the current number — a fresh database never replays the migration
history. Version 0 is not an old schema and never runs the released-schema migration runner.
Migrations form an ordered list of steps; each step is keyed by the non-zero `from_version`
it upgrades and the single `to_version` it produces, and the steps run one at a time. The
released `1 -> 2` step adds the nullable `agents.launch_token` column and its partial index
(an index that covers only the rows where the column is set) via `ALTER TABLE ADD COLUMN`,
after confirming the live v1 `agents` shape with `PRAGMA table_info` (never by matching error
text); a crash before this migration commits leaves the database at version 1 with no
half-created objects. The released `3 -> 4` step adds
`tasks.worktree_path`/`worktree_branch`/`worktree_base_ref` and the new `review_worktrees`
table. Their all-or-none pairing `CHECK` spans three columns, and SQLite's
`ALTER TABLE ADD COLUMN` cannot add a `CHECK` that mentions another column — so `tasks` is
rebuilt rather than altered in place: copy the rows to a holding table, `DROP` the old table,
re-`CREATE` it under the final name with the released v4 SQL, copy the rows back, and
recreate its indexes. Before doing so, the step confirms the live v3 `tasks` shape by
comparing its full SQL — reduced to one standard form so the two texts can be compared, not
just the column names — against the released v3 text. The released `4 -> 5` step creates the
`agent_mutations` table and its three `agents` triggers, and drops the redundant
`idx_task_events_task`; no table is rebuilt. Its validation compares the live index's SQL, in
the same standardized form, against the released v4 text (the index is the only object this
step destroys) and refuses to run if stray objects already occupy the new v5 names. The
released `5 -> 6` step extends the `messages.kind` CHECK with the structured `clear_safe`
Sign-off value (ADR-0016): a CHECK cannot be changed in place, so `messages` is
rebuilt with the same copy/DROP/re-CREATE/refill procedure and the same full-SQL validation
as the earlier destructive rebuilds, recreating its four indexes. The released `6 -> 7` step
creates `observable_mutations` and its nine Message, Task, and Task Event triggers; before
applying the released SQL it refuses to run if stray objects already occupy the new v7
names. The `user_version` stamp on the last line of the binding SQL block above always
equals the implementation's `CURRENT_SCHEMA_VERSION`; a test asserts the two can never drift
apart. Once several schema versions have shipped, at least the two most
recently released versions can be migrated. Anything older receives an instruction to export
and upgrade, rather than a best-effort attempt to change it in place.

## Retention and deletion

- `prune --messages-before <duration>` deletes only Messages that are already read and older
  than the cutoff (`read_at IS NOT NULL AND created_at < now − seconds`, using strict `<`).
  The default is 30 days.
- `prune --tasks-before <duration>` deletes only completed Tasks older than the cutoff
  (`completed_at < now − seconds`, strict `<`) whose every linked Message is already read.
  The default is 90 days.
- Durations are `<integer><s|m|h|d|w>`. A `0`, a fraction, a compound form (`1d12h`), or a
  value so large that `now − seconds` would leave the range of safely representable integers
  is rejected as `USAGE`.
- A single `BEGIN IMMEDIATE` transaction deletes rows in an explicit order that respects the
  references between tables, so SQLite's `changes()` count never includes rows removed by
  automatic cascading deletes and the reported counts are exact: (1) the Task Events of
  eligible Tasks, (2) the Messages linked to eligible Tasks, (3) the eligible Task rows
  themselves, (4) the remaining old read Messages. `messages_deleted` adds up steps 2 and 4;
  `tasks_deleted` is step 3.
- `--vacuum` runs `VACUUM` (SQLite's rebuild of the file that reclaims disk space) outside
  any transaction, only after the prune has committed, and only when no active Agents exist;
  otherwise the already-committed `prune_result` is still emitted, with `vacuumed:false`, and
  the command exits 1 (`ACTIVE_AGENTS`).
- Agent rows are kept until a full `clean`; that way every old Message and Task still points
  at a real Agent.
- `clean` removes `crew.db` and its `-wal` and `-shm` helper files, and leaves all tracked
  configuration untouched. Without `--force` it refuses while active Agents exist — or when
  the store cannot even be opened to count them. `clean --force` never opens the database at
  all — it is the recovery path for a corrupt or locked store — and silently skips files that
  are already gone.

## Health checks

`doctor` opens its own **read-only** diagnostic connection (`DatabaseSync` with
`readOnly:true`, extensions denied), so diagnosing a broken store can never initialize it,
migrate it, or write to its write-ahead log. The Store layer only reports raw facts; the
`doctor` command decides which finding code and severity each fact maps to. The read-only
checks cover:

- that `user_version` is supported (a newer version, or a version-0 file that is not empty,
  → `UNSUPPORTED_SCHEMA`);
- SQLite's `PRAGMA quick_check` and `foreign_key_check` consistency checks (a failure →
  `INTEGRITY`);
- that the expected tables, indexes, and constraints are all present (any divergence →
  `SCHEMA_DRIFT`);
- expired Leases (`STALE_LEASE`) and unfinished Tasks whose assignee or reviewer is archived
  (`ARCHIVED_OWNER`);
- that the state path is where it should be and its parent directory is writable
  (`STATE_PATH`), a best-effort check for a network filesystem where one can be detected
  (`NETWORK_FILESYSTEM`), and a missing store (`NO_STATE_STORE`);
- `tmux` or `git` missing from PATH (`DEPENDENCY_MISSING`), and edited copies of the built-in
  Role/Team seed files (`ROLE_DRIFT`/`TEAM_DRIFT`).
- One check is **skipped in read-only diagnosis**: whether a normal writable open ends up in
  WAL mode, because that cannot be established without writing.
